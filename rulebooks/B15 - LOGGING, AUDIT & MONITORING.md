## 1. PURPOSE

This rulebook defines **logging, audit, and monitoring implementation standards** for YSS Orbit.

It establishes:
- System observability
- Audit trail requirements (including before/after state capture)
- Application vs audit log separation
- Monitoring and alerting
- Incident detection

All system activity MUST be traceable and observable.

---

## 2. SCOPE

Applies to: all API requests and responses, all background jobs, all database operations, all security-related events. No operation is exempt.

---

## 3. TWO SEPARATE LOGGING SYSTEMS (MANDATORY)

These are SEPARATE systems with different purposes. Mixing them is PROHIBITED.

| System | Purpose | Storage | Retention |
|--------|---------|---------|-----------|
| **Audit Log** | WHO did WHAT - compliance, accountability | Database (immutable, append-only) | Per compliance policy (C03) |
| **Application Log** | System health, debugging, errors | Log files / APM (Datadog, Grafana, ELK) | Short-term (30–90 days) |

---

## 4. CORE GOVERNANCE LAWS

### 4.1 Request Logging (MANDATORY)

Every request MUST be logged with the enriched standard structure:

```json
{
  "timestamp": "2025-06-15T10:30:00.000Z",
  "level": "INFO",
  "service": "yss-orbit-api",
  "module": "payroll",
  "correlation_id": "uuid",
  "trace_id": "uuid",
  "request_id": "uuid",
  "user_id": "uuid",
  "business_unit_id": "uuid",
  "endpoint": "/api/v1/payroll/runs/",
  "method": "POST",
  "status_code": 202,
  "duration_ms": 45,
  "message": "Payroll run initiated"
}
```

**CRITICAL DISTINCTION - all engineers MUST know:**

| Field | What it tracks | Generated by | Lifespan |
|-------|---------------|--------------|----------|
| `correlation_id` | One complete user action (may span many requests, jobs, events) | API middleware on first request | Until all async work completes |
| `trace_id` | One APM span (single function/DB call) | OpenTelemetry auto-instrumentation | Single span duration |
| `request_id` | One HTTP request | Per-request middleware | Single HTTP request |

When debugging: search by `correlation_id` to find ALL related logs, jobs, and events.

### 4.2 Audit Logging with Before/After State (MANDATORY)

All critical data mutations MUST include `old_values` and `new_values`.

Required audit log structure:

```python
{
    "correlation_id": "uuid",   # MANDATORY: business operation ID (see E03)
    "trace_id": "uuid",          # APM span ID (OpenTelemetry)
    "timestamp": "utc_iso8601",
    "user_id": "uuid",               # Opaque UUID ONLY - no PII
    "business_unit_id": "uuid",
    "action": "INVENTORY_UPDATE",    # MODULE_ACTION format (UPPER_SNAKE_CASE)
    "resource": "Inventory",
    "resource_id": "uuid",
    "old_values": {...},             # State BEFORE mutation - REQUIRED for UPDATE/DELETE
    "new_values": {...},             # State AFTER mutation - REQUIRED for CREATE/UPDATE
    "ip_address": "sanitized",
    "status": "SUCCESS"              # or "FAILURE"
}
```

Rules:
- `old_values` MUST be captured BEFORE any write operation
- `new_values` MUST be captured AFTER successful write
- `old_values` and `new_values` MUST exclude sensitive fields (passwords, tokens, raw PII)
- Audit log write MUST be fire-and-forget - failure MUST NOT crash the main operation

Fail-Safe Audit Write Pattern (MANDATORY):

```python
def write_audit_log(action, resource_id, old_values, new_values, **kwargs):
    try:
        AuditLog.objects.create(
            action=action,
            resource_id=resource_id,
            old_values=old_values,
            new_values=new_values,
            **kwargs,
        )
    except Exception as e:
        logger.error(f"Audit log write failed: {e}")
        # NEVER re-raise - audit failure must NOT fail the business operation
```

**Violation:** Missing `old_values`/`new_values` on mutations = compliance gap (HIGH).


### 4.13 Event & Orchestrator Logging (MANDATORY)

All domain event publications MUST be logged:

```json
{
  "timestamp": "ISO8601",
  "level": "INFO",
  "type": "DOMAIN_EVENT_PUBLISHED",
  "event_type": "attendance.finalized",
  "event_id": "uuid",
  "event_version": "1.0",
  "aggregate_id": "uuid",
  "business_unit_id": "uuid",
  "correlation_id": "uuid",
  "outbox_id": "uuid"
}
```

All event consumer executions MUST be logged:

```json
{
  "timestamp": "ISO8601",
  "level": "INFO",
  "type": "EVENT_CONSUMED",
  "event_type": "attendance.finalized",
  "event_id": "uuid",
  "consumer": "payroll.handle_attendance_finalized",
  "status": "succeeded",
  "duration_ms": 234,
  "retry_count": 0,
  "business_unit_id": "uuid",

```

All orchestrator steps MUST be logged:

```json
{

  "level": "INFO",
  "type": "ORCHESTRATOR_STEP",
  "orchestrator": "PayrollOrchestrator",
  "workflow_id": "uuid",

  "status": "completed",
  "entity_id": "uuid",
  "business_unit_id": "uuid",
  "correlation_id": "uuid",

}
```

### 4.3 Security Events (MANDATORY)

The following events MUST ALWAYS be logged:
- Failed login attempts
- Successful login and logout
- Account lockout
- Token misuse or invalid token
- Unauthorized access attempt (403)
- Cross-tenant access attempt
- Permission denied
- Password reset events
- Break-glass access activation
- Module subscription check failed (tenant tried to access unsubscribed module)
- Plan limit exceeded attempt
- Feature flag access denied

Log structure for module access denied:
```json
{
  "type": "MODULE_ACCESS_DENIED",
  "module_code": "PAYROLL",
  "business_unit_id": "uuid",
  "user_id": "uuid",
  "correlation_id": "uuid",
  "timestamp": "ISO8601"
}
```

### 4.4 Traceability (MANDATORY)

- Every operation MUST include a unique `trace_id` AND `correlation_id`
- `correlation_id` MUST propagate across ALL layers including background jobs and domain events
- `trace_id` MUST propagate across all services, repositories, background tasks, and log entries
- Logs MUST be correlated by `correlation_id` for cross-system debugging

### 4.5 Log Structure (MANDATORY)

- All logs MUST be JSON-structured
- Unstructured (plain text) log lines are PROHIBITED
- All logs MUST use UTC timestamps

### 4.6 Data Protection in Logs (MANDATORY)

Logs MUST NOT contain:
- Passwords or password hashes
- Access or refresh tokens
- CSRF tokens
- Full personal data (names, emails, phone numbers)
- Payment card data
- Any data classified as RESTRICTED (per C02)

Logs MUST use opaque identifiers:
- `user_id` (UUID) instead of email or name
- `resource_id` (UUID) instead of sensitive field values

### 4.7 Log Immutability (MANDATORY)

- Audit logs MUST be append-only
- Modification or deletion of audit log records is PROHIBITED
- Log tampering is a CRITICAL violation

### 4.8 Log Storage (MANDATORY)

- Application logs MUST be stored centrally (not local-only)
- Audit logs MUST be stored in the database as the primary compliance record
- Recommended: external APM for application logs (Datadog, Grafana, ELK)

### 4.9 Monitoring (MANDATORY)

System metrics MUST be monitored:
- CPU and memory usage
- API response times and latency percentiles
- Error rates
- Task queue depth and failure rates
- Cache hit/miss rates

### 4.10 Alerts (MANDATORY)

Alerts MUST be configured for:
- System failures
- High error rates (> 5% errors over 5 minutes)
- Security incidents (failed login spikes, unauthorized access patterns)
- Slow query threshold breaches (> 500ms)
- Failed background tasks
- Dead-letter queue growth

### 4.11 Retention Policy (MANDATORY)

- Audit logs MUST be retained per compliance requirements (C03)
- Automatic deletion of audit logs is PROHIBITED without compliance review
- Application logs MAY be retained for shorter periods (30–90 days)

### 4.12 Background Task Logging (MANDATORY)

All async tasks MUST be logged with:
- Task name
- Input parameters (sanitized - no sensitive data)
- Execution status (started, succeeded, failed)
- Retry count
- Duration (ms)
- `correlation_id` (MANDATORY - propagated from triggering request, see E03 §3.1)
- `trace_id` (APM span ID)
- `business_unit_id` where applicable

---

## 5. SECURITY & COMPLIANCE

- Audit logs MUST be complete, accurate, and tamper-proof
- Every action MUST be attributable to `user_id` or system actor
- Anonymous critical actions are PROHIBITED
- Audit log deletion = CRITICAL violation

---

## 6. NON-NEGOTIABLE RULES

- Missing logs for critical operations = PROHIBITED
- PII in logs = CRITICAL violation
- Missing `trace_id` = PROHIBITED
- Missing `correlation_id` = PROHIBITED
- Plain text (non-JSON) log lines = PROHIBITED
- Audit log tampering = CRITICAL violation
- Audit log failure crashing main operation = PROHIBITED
- Mixing audit events with application debug logs = PROHIBITED
- Missing `old_values`/`new_values` for mutations = HIGH violation

---

## 7. VIOLATIONS & ENFORCEMENT

| Severity | Action |
|---------|--------|
| CRITICAL | Block deployment |
| HIGH | Reject changes |
| MEDIUM | Fix required |

---

## 8. TESTING REQUIREMENTS

- Logging MUST be tested for all critical flows
- Audit logs MUST be validated for `old_values` / `new_values` correctness
- Fail-safe audit write behavior MUST be tested
- Monitoring alerts MUST be tested
- Log sanitization MUST be verified (no PII)
- `trace_id` propagation MUST be tested end-to-end
- Any failing test MUST block deployment

---

## 9. QUICK SUMMARY

- Audit logs and application logs are SEPARATE systems
- All mutations MUST log `old_values` and `new_values`
- Audit log write MUST be fire-and-forget
- All actions MUST be logged with `trace_id`
- Logs MUST be secure, immutable, and PII-free
