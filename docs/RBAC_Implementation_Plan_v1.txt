1. Architecture — before & after
1.1 Current state (dual-pane)
The existing 
RolesPage.tsx
 renders a two-column layout: BU selector pane on the left, role/permission detail pane on the right. Problems identified:
Four columns squeezed into a single view — unreadable on anything below 1280 px.
Permissions list overflows its column and requires horizontal scroll, clipping permission keys mid-word.
No URL routing — refreshing the page loses your position.
No unsaved-changes protection when switching BUs or roles.
BU scope isolation not visually enforced — easy to mistake which BU a role belongs to.
1.2 New state (drill-down state machine)
RolesPage.tsx
 becomes a route-controlled state machine rendering one of three full-width screens:
Screen
Component
Route
Condition
Screen 1
BUGridScreen
/rbac
No BU selected
Screen 2
RoleListScreen
/rbac/:buId
BU selected, no role selected
Screen 3
PermissionEditorScreen
/rbac/:buId/roles/:roleId
BU + role both selected
Why URL routing instead of component state?
Using React Router means: (1) browser Back button works correctly across all screens, (2) users can bookmark or share a deep link to a specific role, (3) breadcrumbs always match the actual URL — no sync bugs between URL and displayed state, (4) page refresh does not drop you back to Screen 1.

2.2 RolesPage.tsx — route controller
This file is now a thin shell. Its only job is to read route params and render the correct screen. All data fetching and business logic moves into the screen components and hooks.
// RolesPage.tsx
// PURPOSE: Route controller — reads URL params, renders correct screen.
// IMPORTANT: Contains NO data fetching, NO business logic.
// IMPORTANT: Uses <Routes> nested inside the parent /rbac route.
export const RolesPage: React.FC = () => (
  <Routes>
    <Route index                         element={<BUGridScreen />} />
    <Route path=":buId"                  element={<RoleListScreen />} />
    <Route path=":buId/roles/:roleId"    element={<PermissionEditorScreen />} />
  </Routes>
);

5.3 Dirty state & unsaved changes guard
This is the most important non-visual requirement for Screen 3. Without it, an admin can toggle 20 permissions, accidentally click the breadcrumb, and lose all their work silently.
// usePermissions.ts
// Manages local permission state and tracks whether it differs from server state.
// STATE:
//   serverPerms: Record<string, boolean>   — what the API returned
//   localPerms:  Record<string, boolean>   — what the user has toggled
//   isDirty: boolean                        — localPerms !== serverPerms
// isDirty is computed by comparing localPerms to serverPerms key-by-key.
// Revert button resets localPerms to serverPerms and clears isDirty.
// UnsavedGuard.tsx:
//   Uses useBlocker() from React Router v6.8+ to intercept navigation.
//   When isDirty=true and user tries to navigate away:
//     - Show confirmation modal: "Discard unsaved changes?"
//     - "Stay" button: cancels navigation
//     - "Discard" button: allows navigation, resets local state
//   IMPORTANT: Blocker must be reset on successful save.
// Save flow:
//   1. User clicks "Save changes"
//   2. PATCH /api/v1/iam/business-units/{buId}/roles/{roleId}/permissions/
//      Body: { permissions: Record<string, boolean> }
//   3. On success: update serverPerms = localPerms, isDirty = false, show toast
//   4. On error: show error toast, keep localPerms unchanged, do NOT revert

{

8. Transitions & animations
Keep animations subtle and fast. This is an enterprise admin tool — transitions help orientation but must never feel slow or decorative.
Event
Animation
Notes
Screen enter (drill-down forward)
fadeIn + slideInRight
150 ms ease-out. New screen slides in from the right.
Screen exit (back navigation)
fadeOut + slideOutLeft
150 ms ease-in. Old screen slides out to the left.
Permission toggle
toggle thumb slide
150 ms CSS transition on transform. Background colour transitions 100 ms.
Card hover (Screen 1)
border-color transition
100 ms. No scale, no shadow — flat design only.
Scope banner
No animation
Always visible, always static. Animating a warning banner undermines trust.
Save bar
No animation
Always in flow at bottom. Never sticky/fixed.
Respect prefers-reduced-motion
Wrap all CSS keyframe animations in @media (prefers-reduced-motion: no-preference) { ... }. When the user has reduced motion enabled, transitions should be instant (0 ms) — not slower or replaced with fades. Remove the animation entirely.

10. Verification checklist
The implementation is complete when every item below is ticked. Use this as the acceptance criteria for PR review.
10.1 Navigation & routing
Navigating to /rbac renders BUGridScreen.
Clicking a BU card navigates to /rbac/:buId and renders RoleListScreen.
Clicking a role row navigates to /rbac/:buId/roles/:roleId and renders PermissionEditorScreen.
Browser Back button returns to the previous screen correctly at every level.
Refreshing the page at /rbac/:buId/roles/:roleId renders Screen 3 directly (no redirect to Screen 1).
Bookmarking and revisiting a deep link works.
10.2 Scope isolation
ScopeBanner is visible on Screen 2 and Screen 3 at all viewport widths.
ScopeBanner text correctly names the current BU.
Roles from BU-A never appear on BU-B's Screen 2.
Permissions from BU-A never appear on BU-B's Screen 3.
10.3 Unsaved changes guard
Toggling any permission sets isDirty = true.
Clicking breadcrumb or Back while isDirty shows the "Discard changes?" modal.
"Stay" cancels navigation, local state preserved.
"Discard" navigates away, local state reset to server state.
Successful save clears isDirty = false, blocker is released.
10.4 Permissions & RBAC gates
rbac.view only: all screens render, all toggles are disabled, Save button replaced with "View only" message, Assign button hidden.
rbac.manage: all toggles enabled, Save button visible, Assign button visible.
System roles: toggles enabled, role name read-only, Delete button disabled with tooltip.
10.5 Loading, error & empty states
Screen 1: skeleton cards while loading, error alert on failure, empty state when no BUs exist.
Screen 2: skeleton rows while loading, error alert on failure, empty state when no roles exist with "Add first role" CTA.
Screen 3: skeleton toggles while loading, error alert on failure.
10.6 Accessibility
All interactive elements (BU card, role row, permission toggle) are keyboard-reachable via Tab.
Permission toggles have role="switch" and aria-checked.
BU cards have role="button" and respond to Enter/Space.
Breadcrumb uses <nav aria-label="Breadcrumb"> wrapping an <ol>.
Scope banner has role="alert" so screen readers announce it on screen change.
All icon-only buttons have aria-label.
@media (prefers-reduced-motion: no-preference) applied to all animations.
10.7 Responsive
BU grid is 1 column on mobile, 2+ on tablet, 3+ on desktop.
Permission grid is 1 column on mobile, 2+ on desktop.
Breadcrumb truncates on < 768 px without overflow or horizontal scroll.
Save bar does not overflow or clip on any viewport width.
