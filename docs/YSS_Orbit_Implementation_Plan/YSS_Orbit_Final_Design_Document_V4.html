Executive Summary — Platform Snapshot

Architecture Style
Core Platform + App Store Subscription Architecture
Domain-driven design with bounded contexts. Deployable as monolith today, decomposable into services without rewrites.
Tenancy Model
Shared Database, Row-Level Isolation
Single PostgreSQL cluster. Every table has business_unit_id. Repository layer enforces isolation automatically on every query.
Core Stack
Django · DRF · PostgreSQL · Redis · Celery
REST API (v1 versioned). Redis for caching, sessions, and queues. BullMQ for background jobs and event processing.
Security Model
JWT + RBAC + Module Gate + Tenant Filter
4-layer request security: Auth → RBAC → Module subscription check → Automatic tenant data filter. Backend is the only security boundary.
Scalability Strategy
Horizontal scaling + Read replicas
Stateless API servers behind load balancer. Primary DB for writes. Read replicas for reports. Redis cluster for scale. Table partitioning for high-volume data.
Deployment Strategy
Containerized · Multi-region ready
Docker containers orchestrated via Kubernetes. Blue-green deployments. Zero-downtime migrations. CDN for static assets.
Event Architecture
Outbox Pattern + Domain Events
Events written atomically to outbox table. Worker reliably publishes to event bus. Domain events versioned. Dead-letter queues for failures. Replay supported.
RBAC Summary
Role → Permission → Module-scoped
Roles: Super Admin, Org Admin, BU Admin, HR Manager, Employee, and custom. Permissions are granular per action. Users carry one global identity across all tenants they belong to.

Architecture Decision
Start as a modular monolith. The bounded context separation means any module can be extracted to a microservice later without data model changes. Don't prematurely distribute.

Platform Definition & Core Architecture

YSS Orbit is a Multi-Tenant Modular Enterprise SaaS Platform. One platform serves multiple industries, multiple client companies, multiple branches, and multiple modules — all within a single centralized, isolated system. One codebase, one database cluster, infinite tenants — each isolated by business_unit_id.
Platform Definition
What it is
A single hosted platform serving unlimited client companies across different industries, each getting their own isolated tenant environment with their own modules, users, data, and settings — none visible to others.
What it is NOT
Not a single-client CRM. Not a per-client codebase. Not a monolith deployed once per customer. The same running service handles Retail, Pharmacy, and HRMS clients with total data isolation between every branch.
Core Architecture Hierarchy
🌐

🏭

⚙️

Architecture Law #1
BusinessUnit = Tenant. Not the Domain, not the Organization. The BusinessUnit is the exact boundary where data isolation is enforced. Every operational table carries business_unit_id NOT NULL. Every repository query auto-appends WHERE business_unit_id = ?.

Business Domains

A Business Domain is not a tenant and not an industry label only. It defines the entire operational ecosystem — workflows, module templates, and business logic groupings. Domain suggests default modules during onboarding but never restricts which modules a tenant can subscribe to.
Critical Distinction — Domain vs Module vs Tenant
Most Common Confusion
Domain ≠ Module ≠ Tenant. Domain = business ecosystem category (what kind of business). Module = functional capability (what the software does). Tenant = BusinessUnit (where the data isolation happens). A Retail client can and should enable HRMS, Payroll, and Attendance modules. Domain never locks module access.

// Tenant Boundary

BusinessUnit — The Real Tenant

The BusinessUnit is the most important concept in the platform. It is the exact data isolation boundary. Every employee, payroll record, attendance log, invoice, inventory item, report, and user permission is scoped to a BusinessUnit. Nothing crosses BusinessUnit boundaries without explicit authorization.
Isolation Law
NEVER Violate
Reliance Vizag can NEVER see Reliance Hyderabad data.
Apollo Guntur can NEVER see Apollo Vizag data.
Infosys Hyderabad can NEVER see Infosys Bangalore data.

Even within the same Organization, each BusinessUnit is a fully isolated data container. Enforced at the database query level via WHERE business_unit_id = ? on every single query — not in the frontend.

// Module System

Module Registry & Tenant Module Subscriptions

The Module Registry is the programmatic gatekeeper controlling which modules each tenant can access. Without it there is no SaaS — just a single app. Every module API call is gated by middleware that checks business_unit_module. If the tenant has not subscribed to the module, the request is rejected with 403.
Module Master Table
-- module table (Platform-owned)
id               UUID PRIMARY KEY
code             VARCHAR(50) UNIQUE NOT NULL  -- 'HRMS', 'POS', 'PAYROLL'
name             VARCHAR(100) NOT NULL
description      TEXT
icon             VARCHAR(50)
depends_on       UUID[]                       -- module IDs this requires first
is_active        BOOLEAN DEFAULT TRUE
version          VARCHAR(20) DEFAULT '1.0.0'
created_at       TIMESTAMP DEFAULT NOW()
Tenant Module Subscription Table
-- business_unit_module table
id               UUID PRIMARY KEY
business_unit_id UUID REFERENCES business_units(id) NOT NULL
module_id        UUID REFERENCES module(id) NOT NULL
status           ENUM('active','trial','suspended','expired')
plan_limit       JSONB          -- {max_users: 50, max_records: 10000}
trial_ends_at    TIMESTAMP
activated_at     TIMESTAMP
expires_at       TIMESTAMP      -- NULL = no expiry
activated_by     UUID REFERENCES users(id)
created_at       TIMESTAMP DEFAULT NOW()
UNIQUE(business_unit_id, module_id)
Module Access Check — Middleware Flow
Request arrives
→
Extract business_unit_id from SecurityContext
→
Query business_unit_module WHERE bu_id=? AND module=? AND status='active'
→
Not found → 403 Module Not Subscribed
→
Found → proceed to handler
Implementation Rule
Module access check runs after authentication and after RBAC. Order: Auth → RBAC → Module Check → Service Layer. Cache module subscription status in Redis with key module_access:{bu_id}:{module_code} with 5-minute TTL to avoid hitting DB on every request.

// Domain-Driven Design

Domain-Driven Architecture & Bounded Contexts

Each module is a Bounded Context in Domain-Driven Design terms. They have their own models, services, and repository layers. Cross-module communication happens through domain events — never through direct service-to-service calls. This prevents tight coupling and allows modules to scale independently.
Bounded Contexts
HR Context
Employee Aggregate
Employee is the root aggregate. All employee-related data lives here. Other contexts reference employee by ID only — never join across contexts.
Payroll Context
PayrollRun Aggregate
Payroll run is the root. Consumes employee.id and attendance summaries via events. Never directly queries the HR database.
Inventory Context
StockItem Aggregate
Stock item and batch are roots. POS context references item.id. Stock levels updated via stock transaction events.
Billing Context
Invoice Aggregate
Invoice is the root aggregate. References customer, items, and payment records. GST computations isolated here.
DDD Rule
No context directly queries another context's database tables. The Payroll module never runs a JOIN against the attendance table. It consumes a summarized attendance record via domain event or a read-model built specifically for payroll computation. This keeps bounded contexts decoupled and independently deployable.

// Security

Security Architecture & Request Flow

Security is enforced entirely in the backend. The frontend is untrusted. Every API request passes through a strict pipeline before touching business logic. The SecurityContext object carries the verified identity and tenant scope for the entire request lifecycle.
Complete Security Pipeline
STEP 1
Incoming Request — POST /api/v1/payroll/run · JWT in Authorization header
STEP 2
Authentication — Verify JWT signature, check expiry, extract user_id. Invalid → 401 Unauthorized.
STEP 3
RBAC Middleware — Load user's role and permissions for requested business_unit_id. Check required permission (payroll.create). Missing → 403 Forbidden.
STEP 4
Module Subscription Check — Verify tenant has active subscription. Not subscribed → 403 Module Not Subscribed.
STEP 5
SecurityContext Built — {user_id, business_unit_id, allowed_bu_ids, roles, permissions, active_module, correlation_id} injected into request.
STEP 6
Service Layer — Receives SecurityContext. All business logic uses ctx.business_unit_id. Never trusts request body for tenant identity.
STEP 7
Repository Layer — Auto-appends WHERE business_unit_id = ctx.business_unit_id on ALL queries. Developers cannot skip it.
STEP 8
Response — Tenant-filtered data returned. Audit log entry written with correlation_id.
SecurityContext Object
# SecurityContext — injected into every service method
{
  "user_id":                   "uuid",
  "business_unit_id":         "uuid",     // active tenant
  "allowed_business_unit_ids": ["uuid1", "uuid2"],
  "organization_id":          "uuid",
  "roles":                     ["HR_MANAGER"],
  "permissions":               ["payroll.create", "employee.view"],
  "active_module":             "PAYROLL",
  "correlation_id":            "uuid",    // ties request → audit → logs → jobs
  "session_id":               "uuid",
  "ip_address":               "string"
}
Security Law — NEVER Violate
NEVER TRUST THE FRONTEND. The business_unit_id governing data access must come from the verified SecurityContext — never from the request body, query params, or headers sent by the client. A malicious user changing business_unit_id in the frontend must be impossible to exploit.

Enterprise Database Patterns
Outbox Pattern
All domain events are written to an outbox table inside the same transaction as the business operation. A background worker reliably publishes them to the event bus. This guarantees events are never lost even if the broker is temporarily down.
Read Replica Strategy
All reporting queries, dashboard aggregations, export jobs, and analytics are routed to read replicas. The primary database handles only writes and real-time reads. This prevents heavy reports from slowing operations.
Table Partitioning
High-volume tables — attendance_logs, payroll_runs, audit_log, stock_transactions — are partitioned by month/year. Old partitions are archived without affecting live query performance.
Idempotency Keys
All financial APIs (payroll run, invoice create, payment record) accept an idempotency_key header. Duplicate requests with the same key return the original response without re-processing. Prevents double payroll runs from network retries.

API Versioning & URL Standards

All APIs must be versioned from day one. Once a single client integrates with an unversioned endpoint, you can never safely refactor it. The versioning prefix costs zero effort to add now and saves weeks of backward-compatibility work later.
URL Structure Standard
# Standard: /api/v1/{module}/{resource}/{id?}/{action?}

GET    /api/v1/hrms/employees
POST   /api/v1/hrms/employees
GET    /api/v1/hrms/employees/{id}
PUT    /api/v1/hrms/employees/{id}
DELETE /api/v1/hrms/employees/{id}

POST   /api/v1/payroll/runs
GET    /api/v1/payroll/runs/{id}
POST   /api/v1/payroll/runs/{id}/approve
GET    /api/v1/payroll/runs/{id}/payslips

POST   /api/v1/pos/bills
POST   /api/v1/pos/bills/{id}/refund

GET    /api/v1/inventory/items
POST   /api/v1/inventory/stock-transactions
GET    /api/v1/inventory/reports/stock-valuation

# Platform/admin
GET    /api/v1/platform/business-units
POST   /api/v1/platform/business-units/{id}/modules/activate
GET    /api/v1/jobs/{job_id}               -- background job status polling
Standard Response Format
// Success
{ "success": true, "data": {...}, "meta": {"page":1,"per_page":20,"total":150}, "timestamp": "ISO8601" }

// Error
{ "success": false, "error": { "code": "PAYROLL_ALREADY_PROCESSED", "message": "...", "field": "month" } }
Versioning Rule
Never expose /api/ without a version in production. When v2 is needed, run v1 and v2 in parallel. Add a Deprecation response header to v1 endpoints 3 months before removal. Mobile apps may be on old versions for months — breaking them costs real clients real money.

Orchestration Services

Complex multi-domain workflows execute through Orchestration Services instead of direct module-to-module service calls. An Orchestrator owns the workflow state, coordinates calls to individual bounded contexts, handles failures and compensations, and publishes the final domain event. This is the enterprise pattern that prevents spaghetti module dependencies.
Core Orchestrators
PayrollOrchestrator
Coordinates: (1) fetch attendance summary, (2) fetch leave deductions, (3) apply salary components, (4) compute TDS/PF/ESI, (5) generate payslips, (6) publish payroll.generated event, (7) trigger email payslips job.
EmployeeLifecycleOrchestrator
Handles onboarding: (1) create employee record in HRMS, (2) set up payroll component template, (3) set leave balance defaults, (4) provision user account if needed, (5) send welcome notification.
InventoryTransferOrchestrator
Manages inter-branch transfers: (1) validate source stock, (2) create transfer request, (3) approval workflow, (4) deduct source inventory, (5) credit destination inventory, (6) publish transfer event.
SubscriptionOrchestrator
Handles plan changes: (1) validate new plan limits, (2) update subscription record, (3) enable/disable modules per new plan, (4) update feature flags, (5) notify tenant admin, (6) write audit log.
OffboardingOrchestrator
Employee exit: (1) final payroll settlement, (2) revoke system access, (3) clear leave balances, (4) generate experience letter job, (5) archive employee record (soft delete), (6) compliance audit entry.
ProcurementOrchestrator
Purchase order flow: (1) validate vendor, (2) create PO, (3) approval routing, (4) GRN processing, (5) inventory credit, (6) payable entry in billing, (7) notify store manager.

Workflow Rule
Complex workflows must NEVER execute through chained direct module service calls. Service A calling Service B which calls Service C creates untraceable coupling and impossible-to-retry failures. // Non-Negotiables

Enterprise Non-Negotiable Rules

These are the rules that define YSS Orbit as an enterprise platform. None can be compromised for speed, convenience, or client pressure. Breaking any of these turns the platform into a liability.
Rule 1 — Tenant Isolation
BusinessUnit is ALWAYS the tenant boundary. Every query against an operational table must filter by business_unit_id. The Repository layer enforces this automatically on every query. No developer can query without it.
Rule 2 — Never Trust Frontend
Backend is the only security boundary. Frontend can show/hide buttons. Backend enforces every permission, every tenant filter, every plan limit. A user manipulating the frontend cannot bypass backend controls.
Rule 3 — API Versioning from Day One
All APIs are /api/v1/ from the very first endpoint. No unversioned routes in production. Version before any client integrates — changing it later breaks every existing integration.
Rule 4 — User Identity is Global
One email = one User account, always. Multiple tenant accesses use UserBusinessUnit mapping. No duplicate accounts for the same person across different clients.
Rule 5 — Domain Does Not Restrict Modules
Any tenant can subscribe to any combination of modules. Domain is a classification, not a lock. A pharmacy client enabling HRMS is valid. Never reject a module subscription based on domain.
Rule 6 — One Employee, All Modules
One employee record, used by all modules. The employee in HRMS is the same entity in Payroll, Attendance, and Leave. No duplicate employee records per module.
Rule 7 — Long Operations are Background Jobs
Payroll, exports, bulk imports, email sending — never in the HTTP cycle. Queued as jobs, processed asynchronously. Client polls for status. No HTTP timeouts on heavy operations.
Rule 8 — Audit Log is Forever
The audit_log table is append-only and never deleted. Every state change is recorded with before/after values. This is non-negotiable for enterprise compliance and legal accountability.
Rule 9 — Never Hardcode Tenant Settings
All per-tenant operational rules go in tenant_settings. Timezone, payroll cycle, GST number, attendance grace period — always from the settings service, never hardcoded in business logic.
Rule 10 — Soft Delete, Never Hard Delete
No operational record is permanently deleted. Use is_deleted = TRUE and deleted_at. Payroll, invoices, and employees are legal records that must be preserved.
Rule 11 — Orchestrators for Complex Workflows
Multi-step cross-module workflows are owned by Orchestrators. No chained direct service calls between modules. Orchestrators provide traceability, retry logic, and compensation actions.
Rule 12 — Correlation IDs Everywhere
Every request, job, event, and audit entry carries a correlation_id. Generated at API entry and propagated through every system layer. This is how you debug production issues at enterprise scale.

// Elite-Tier Patterns

Operational Excellence — Resilience & Recovery

Elite enterprise platforms don't just handle the happy path — they are specifically engineered for what happens when things fail. This section covers the eight operational resilience patterns that separate an enterprise platform from a prototype: retry strategies, dead-letter queues, idempotency, outbox, saga orchestration, event replay, disaster recovery, and cache invalidation.
Fig 4 — Infrastructure Topology — HA Setup with Horizontal Scaling
Eight Elite-Tier Operational Patterns
// Pattern 1
Retry Strategy with Exponential Backoff
All job workers and event consumers retry failed operations with exponential backoff + jitter. Attempt 1 → 2s, Attempt 2 → 4s, Attempt 3 → 8s (+ random jitter to prevent thundering herd). Max 3 retries for most operations. Payroll and financial operations retry up to 5 times. After all retries exhausted, event moves to Dead-Letter Queue. Retry state is persisted in Redis — workers that crash mid-retry resume from the correct attempt number.
// Pattern 2
Dead-Letter Queues (DLQ)
Every BullMQ queue has a corresponding dead-letter queue. Events that exhaust all retries land in DLQ instead of being silently dropped. A dedicated DLQ monitor alerts Super Admin via email and in-app notification. Each DLQ entry records: original event payload, failure reason, retry attempt count, timestamps. Super Admin can manually replay individual events from DLQ, bulk-replay by time range, or archive after investigation. Nothing is ever silently lost.
// Pattern 3
Idempotency Keys
All state-changing financial APIs accept an Idempotency-Key header. Server stores key → response mapping in Redis with 24-hour TTL. If the same key is seen again, the cached response is returned without re-executing the operation. This prevents: double payroll runs from network retries, duplicate invoices from client retries, duplicate payments from webhook replays. Event consumers use event_id as their idempotency key — already-processed event IDs are stored in a Redis set and skipped on re-delivery.
// Pattern 4
Transactional Outbox Pattern
Domain events are written to an outbox table inside the same database transaction as the business operation. A dedicated Outbox Worker polls the table every second, publishes unprocessed events to BullMQ, then marks them as published. This guarantees: if the business operation commits, the event will eventually be published (at-least-once). If the broker is down, events queue in the DB. No dual-write problem. The outbox table schema: id, business_unit_id, event_type, payload, status (pending/published), correlation_id, created_at, published_at.
// Pattern 5
Saga Orchestration for Long Workflows
Multi-step cross-module workflows (PayrollOrchestrator, EmployeeLifecycleOrchestrator) implement the Saga pattern with compensation actions. Each step is idempotent and logged. If step 3 of a 6-step payroll run fails, the Orchestrator executes compensation — rolling back the partial state of steps 1–2. The orchestrator_state table persists the current step, inputs, and outputs for each workflow instance. If the server crashes mid-flow, the orchestrator resumes from the last completed step on restart. This makes long-running workflows crash-safe.
// Pattern 6
Event Replay
The outbox table retains all published events for 90 days (configurable). Super Admin can replay events by: time range, event type, business_unit_id, or correlation_id. Use cases: rebuilding a subscriber's state after a bug fix, re-sending failed notifications, reconstructing audit trail after data issue. Replay is idempotent — event_id deduplication prevents double-processing. Replay jobs run as background tasks and are tracked with their own correlation_id for observability. This is critical for production incident recovery without manual data patching.
// Pattern 7
Disaster Recovery Strategy
RTO (Recovery Time Objective): <2 hours. RPO (Recovery Point Objective): <5 minutes. Implementation: PostgreSQL WAL-based streaming replication to a standby in a separate AZ. Automated daily snapshots retained for 30 days. Redis persists to disk (AOF + RDB). Object storage (S3) has cross-region replication enabled. Runbook: (1) Promote standby DB, (2) Update connection string env var, (3) Restart API containers, (4) Verify DLQ drains, (5) Validate tenant data integrity. Quarterly DR drills are mandatory for Enterprise SLA compliance.
// Pattern 8
Cache Invalidation Strategy
YSS Orbit caches at 4 levels: (1) Module subscription status — key module_access:{bu_id}:{module_code}, TTL 5min, invalidated on subscription change. (2) Tenant settings — key settings:{bu_id}, TTL 5min, invalidated on settings update. (3) RBAC permissions — key permissions:{user_id}:{bu_id}, TTL 15min, invalidated on role change. (4) Feature flags — key flags:{bu_id}, TTL 10min, invalidated on flag update. All cache keys use the pattern: {entity}:{scope_id}:{qualifier?}. Cache invalidation events are published as internal domain events so distributed workers also invalidate their local caches.

B22 — Non-Negotiable Rule
Branding NEVER affects business logic. The branding layer controls only presentation, labels, themes, and logos. It must never touch RBAC, security, tenant isolation, payroll calculations, or any business workflow. Branding is config — not code. No tenant-specific forks in the codebase. Ever.

Tenant Customization Governance & Configurable Platform Strategy

The single biggest threat to SaaS scalability is customization chaos — where every enterprise client gradually turns into a bespoke software engagement. YSS Orbit prevents this with a hard boundary between what is configurable (via config) and what is not (business logic forks). Every customization request must be evaluated against this boundary before acceptance.
What Is Configurable vs What Is Not
✓ ALLOWED — Configuration

  • Role & permission sets

Tenant Offboarding & Data Termination
Legal Compliance Rule
Tenant offboarding is a legal process, not a database delete. The offboarding sequence is: (1) Export full tenant data as encrypted ZIP (GDPR/compliance). (2) Apply legal_hold flag if applicable — blocks deletion. (3) Soft-delete organization and all BusinessUnits. (4) Archive all data to cold storage with configurable retention window (default 7 years for payroll/HR data). (5) Hard delete only after retention period expires and legal hold is cleared. No tenant data is ever immediately destroyed.

Platform Operations — Support, Diagnostics & Tenant Health

Enterprise SaaS platforms fail operationally even when engineering is excellent. Support teams need tools to diagnose tenant issues without touching production data directly. This section defines the platform ops capabilities: impersonation, tenant diagnostics, health dashboards, support audit logs, and zero-downtime migration strategy.
Support Operations Capabilities
// Impersonation
Super Admin Impersonation
Super Admin can impersonate any user within any BusinessUnit to reproduce reported issues. Every impersonation session is audit-logged with start time, end time, acting admin, and target user. Impersonated actions are marked impersonated: true in all audit entries. Tenant admins can see impersonation events in their own audit log.
// Diagnostics
Tenant Diagnostics Panel
A read-only Super Admin panel showing per-tenant: active users, module subscription status, recent error rates, job queue depth, last payroll run status, feature flag overrides, and DLQ entry count. Diagnosis happens in the dashboard — not in raw SQL against production.
// Health
Platform Health Dashboard
Real-time visibility into: API error rate by tenant, DB connection pool utilization, Redis cache hit ratio, BullMQ queue depths, failed job count, outbox lag, and P95 response times. Threshold breaches auto-alert the on-call engineer via configured channel (Slack, PagerDuty, email).
// Feature Control
Per-Tenant Feature Enablement
Support can enable or disable any feature flag for a specific tenant without a code deployment. This covers beta module access, UI experiments, emergency kill switches, and plan-based feature gates. All flag changes are audit-logged with reason and actor.
Zero-Downtime Migration Strategy

Schema changes and tenant data migrations are the most dangerous operations in a live SaaS platform. YSS Orbit follows the expand-contract pattern:

Expand — add new column/table (backwards compatible)
→
Deploy new code writing to both old and new
→
Backfill data in background job
→
Switch reads to new column
→
Contract — drop old column safely
Migration Rule
No destructive DDL (DROP COLUMN, RENAME, ALTER NOT NULL) is ever run directly on a live production database. All migrations go through the expand-contract sequence. Tenant data backfills run as low-priority background jobs with per-tenant progress tracking. Any migration touching payroll or financial tables requires a second Super Admin approval before execution.

Performance Governance — Reporting, Search & Query Safety

Enterprise platforms accumulate performance debt from three sources: reporting explosion (every client wants custom exports and analytics), N+1 queries (dashboards that fire hundreds of DB calls), and search without indexing strategy (full-table scans as data grows). This section defines the mandatory patterns to prevent all three.
Reporting Governance
Critical Performance Rule
All reports that aggregate more than 1,000 rows MUST run as background jobs — never synchronously. The user requests the report, receives a job ID, and polls for completion. Results are written to object storage (S3) and served as a signed download URL. This prevents analytics load from competing with transactional API traffic.
Report TypeExecution ModeDB Target
Real-time counters / widgetsSynchronous (cached)Redis materialized view
Standard reports (<1K rows)SynchronousRead Replica
Large exports (>1K rows)Background job → S3 downloadRead Replica
Compliance / audit reportsBackground job → S3 downloadArchive DB / cold storage query
Scheduled reports (cron)Background job → emailRead Replica
Global Search Architecture

Enterprise applications become unusable without fast cross-module search. The YSS Orbit search strategy:

// Indexing
Search Index Strategy
A dedicated search_index table (or Elasticsearch/Meilisearch instance for Enterprise tier) is maintained per BusinessUnit. Domain events trigger index updates asynchronously via the event bus. Index entries are scoped to business_unit_id — cross-tenant search is architecturally impossible.
// Entities
Searchable Entities
Employees, payroll runs, attendance records, leave requests, invoices, purchase orders, stock items, audit log entries, and any module-registered entity. Each module registers its searchable fields in the Module Registry metadata.
// Safety
RBAC-Filtered Results
Search results are post-filtered by the requesting user's RBAC permissions. A user without payroll.view permission never sees payroll results in global search — even if they appear in the index. Search never bypasses the permission layer.
Query Safety — N+1 & Cross-Module Join Rules
Query Governance Rule
No cross-domain SQL JOIN is permitted in application code. If a dashboard widget needs data from both the Payroll and Attendance domains, it calls each domain's service separately and aggregates in the application layer. This preserves domain boundaries and prevents one domain's query planner decisions from cascading into another. Cross-domain aggregations that need to be fast use pre-computed materialized summaries updated by domain events.
Anti-PatternCorrect Pattern
N+1: fetch employee list, then query salary for each in loopBatch: single query with WHERE employee_id = ANY($1)
Dashboard fires 20 separate API calls on loadAggregation endpoint returns all widget data in one call (with Redis cache)
JOIN across payroll_runs and attendance_records tablesSeparate service calls; aggregate in application layer
SELECT * on large tables for reportsSELECT only required columns; paginate; run on read replica
Timezone & Payroll Time Governance
Time Governance Rule — B23
All timestamps are stored in UTC. Period. Conversion to local timezone happens only at the presentation layer.
  • Each BusinessUnit has a configured timezone field (IANA format, e.g. Asia/Kolkata).
  • Payroll cut-off dates and attendance shift boundaries are resolved in BU timezone — never in server timezone.
  • Overnight shifts crossing midnight are tracked by shift-start date in BU timezone.
  • DST transitions are handled via date-fns-tz or equivalent — never manual offset arithmetic.
  • Payroll period start/end are stored as DATE (not TIMESTAMPTZ) to avoid DST edge cases on period boundaries.

Permission Taxonomy & Notification Engine Governance

Two systems tend to grow chaotically without governance: permissions (hundreds of overlapping, unnamed, undocumented permissions accumulate) and notifications (modules independently fire emails and in-app alerts without templates, retry, or audit). Both require enforced naming conventions and centralized ownership.
Permission Taxonomy — Naming Standard

Every permission in the platform follows a strict naming convention: domain.resource.action

-- Format: domain.resource.action
payroll.run.create        -- create a payroll run
payroll.run.approve       -- approve a payroll run
payroll.slip.view         -- view payslips
attendance.record.edit    -- edit attendance entries
hrms.employee.create      -- create employee profile
hrms.employee.terminate   -- mark employee as terminated
reports.payroll.export    -- export payroll reports
settings.branding.edit    -- edit tenant brand config
admin.role.assign         -- assign roles to users
Permission Governance Rule
  • Every permission is registered in a central permission_registry table — no undocumented permissions exist.
  • Permissions are grouped into Permission Sets (pre-built role templates) to prevent manual misconfiguration.
  • Deprecated permissions are removed from the registry only after all role assignments using them are migrated.
  • Wildcard permissions (e.g. payroll.*) are supported only for Super Admin — never for tenant roles.
Central Notification Engine Rules

The Notification Engine (Section 16) must be the sole channel for all outbound communications. No module sends email, SMS, or push directly.

RequirementImplementation
Template-drivenAll notification content defined in notification_templates table — no hardcoded strings in module code
Channel abstractionModules emit a NotificationRequested event — the engine decides email / SMS / in-app / push based on tenant preferences
Retry & DLQFailed deliveries retry 3× with exponential backoff; exhausted attempts land in Notification DLQ for ops review
User preferencesEach user can configure per-channel preferences per notification type — engine respects opt-outs
LocalizationTemplates support locale variables; tenant default locale applied if user locale not set
BatchingDigest notifications (e.g. daily attendance summary) are batched in a scheduled job — not fired per-event
AuditEvery sent notification is logged in notification_log with delivery status, channel, template used, and recipient
API & Webhook Platform Governance

Enterprise clients will eventually request API access and webhook integrations. The governance framework must be in place before the first external API key is issued:

// API Keys
Scoped API Keys
API keys are scoped to a specific BusinessUnit and a permission set. Keys are created by Org Admin, displayed once (stored as hash), and can be rotated or revoked at any time. Every API call via key is logged with key ID and BU scope.
// Rate Limits
Per-Key Rate Limiting
Rate limits are enforced per API key in Redis using sliding window. Limits vary by subscription plan. Exceeding rate limit returns 429 Too Many Requests with Retry-After header. Sustained abuse triggers automatic key suspension and ops alert.
// Webhooks
Signed Webhook Delivery
Webhook payloads are signed with HMAC-SHA256 using the tenant's webhook secret. Receiver must verify signature before processing. Delivery is retried up to 5 times with exponential backoff. Failed webhooks land in Webhook DLQ with full payload for manual replay.
// Mobile
Mobile & Offline Strategy
Attendance capture and manager approvals must function offline (mobile-first scenarios). Strategy: local SQLite cache on device, optimistic UI, sync on reconnect with server-side conflict resolution (last-write-wins for attendance, manual resolution flag for approvals). Device registration required for push notifications.

Subscription Engine & File Storage Governance

Two practical SaaS realities that must be architected proactively — not retrofitted: subscription limits enforcement (feature gates, employee caps, storage quotas) and file storage governance (uploads that grow unbounded become a cost and compliance problem). Both require explicit platform-level ownership.
Subscription Engine — Limit Enforcement

The existing Subscription & Plans system (Section 13) and Module Registry (Section 05) provide the foundation. Add enforcement at the API gateway layer:

Limit TypeEnforcement PointBehaviour on Breach

Enforcement Rule
Limit checks are always performed server-side — client-side UI hints are UX only, not security. Limit state is cached in Redis per business_unit_id with a short TTL (60s). Cache is invalidated on subscription upgrade/downgrade events.
File & Document Storage Governance

Every uploaded file (payslips, employee documents, invoices, attendance proof, report exports) must be governed by a lifecycle policy from the moment it is written:

File TypeBucket / PathRetentionArchive After
Payslip PDFspayroll/{bu_id}/{year}/7 years statutory2 years → Glacier

File Storage Rules
  • Virus scanning — every uploaded file passes through ClamAV (or cloud-native equivalent) before being moved to the permanent bucket. Files pending scan are held in a quarantine prefix.
  • No public URLs — all file access is via signed URLs with short expiry (15 min for downloads, 5 min for previews).
  • Storage quotas — enforced per BusinessUnit; uploads rejected with 413 when quota is exceeded.
  • Tenant offboarding — all files are included in the export archive before archival; file deletion follows the same retention policy as data.

YSS Orbit is designed as a scalable, modular, tenant-isolated enterprise SaaS platform supporting multi-domain operations, strict security boundaries, extensible workflows through orchestration, event-driven inter-module communication, full observability with correlation tracing, and long-term enterprise scalability.
